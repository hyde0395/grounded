§01

무엇을 만들었나

기존 가드레일은 전부 "이 명령이 위험한가?"를 묻는다(rm -rf, force push, 시크릿 유출). grounded는 다른 질문을 묻는다 — "이 행동에 근거가 있는가?" 보안 도구가 아니라 근거 강제(grounding enforcement) 도구이며, 보안 가드레일의 대체재가 아닌 보완재다.

핵심 통찰: CLAUDE.md에 "추측하지 마라"라고 쓰는 것은 모델이 런타임에 해석하는 제안이고, 모델은 설득당한다. hook은 모델의 생각과 무관하게 무조건 실행되는 강제다. 근거 없는 행동은 강제로만 막힌다.

규칙강제 내용도입
G-1 읽기 선행이 세션에서 읽지 않은 파일은 Edit/Write 불가. 근거원: Read · Grep(단일 파일) · cat · git diff/show 출력 · 과거 성공한 쓰기v0.1
G-1s 셸 쓰기 검문sed -i · perl -i · tee · > · 미독 파일 위로의 cp/mv → 차단. >>(append)와 런타임 결정 대상(find -exec, xargs)은 경고만v0.2
G-2 존재 확인npm / PyPI / crates.io에 실존하지 않는 패키지는 설치 불가v0.2
G-3 링크 실존404/410/DNS 실패 URL 차단 · 403/5xx/타임아웃은 봇차단 가능성이 있어 경고만v0.3
freshness 신선도읽은 뒤 디스크에서 변경된 파일에 의존하면 "다시 읽어라" 경고v0.5

§03

원래 설계에서 바뀐 것들

명세(grounded-spec.html)는 단일 진실 공급원이었지만, 구현과 라이브 검증이 설계를 다섯 군데에서 고쳐 썼다.

1. G-1의 존재 이유가 재정의됐다 — 가장 큰 설계 수정

E2E 검증에서 Claude Code 내장 read-before-edit 검증이 PreToolUse hook보다 먼저 발화한다는 사실을 발견했다(미독 파일 Edit 시 hook이 아예 안 탐). 즉 G-1의 원래 주력 시나리오는 내장 가드와 중복이었다. G-1의 가치는 재정의됐다: ① 내장 가드가 모르는 근거(cat/grep/git diff) 인정 ② freshness 같은 정밀 규칙의 기반 ③ 내장 가드가 전혀 안 다루는 셸 우회(G-1s). 차별화의 본체는 G-2 · G-3 · G-1s로 이동했고, README 포지셔닝에 그대로 반영했다.

2. 경고는 한 번만 — 멱등화 (설계에 없던 규칙)

원설계의 WARN은 매번 주입이었다. 사용자 피드백("경고 주입이 컨텍스트를 오염시킨다")으로, 반복 경고가 문제 악화처럼 읽혀 모델이 과업 대신 경고를 쫓을 수 있다는 우려가 제기됐다. ledger warned 섹션에 (규칙:대상) 키로 세션당 1회만 주입하고, 모든 경고 뒤에 "advisory · 반복 안 됨 · 지우려고 재시도 금지" 안내를 붙이는 것으로 설계를 바꿨다.

3. ledger는 세션별이 아니라 cwd당 하나

resume 시 session_id가 바뀌면 근거가 증발해 오탐(잘못된 차단)이 난다. 철학이 "오탐 < 누락"이므로 동시 세션 교차 오염(누락)을 감수하고 cwd당 단일 ledger로 결정했다. 같은 이유로 SessionStart의 source가 resume/compact면 ledger를 유지한다 — 대화가 기억하는 근거를 날리면 안 된다.

4. 명세가 경고한 "플러그인 hook 버그"는 재현되지 않았다

명세는 "플러그인 command hook이 차단을 무시한 버그 이력"을 경고했으나 출처가 확인되지 않았다. 대응책(개발은 settings.json 직접 등록 + exit 2 self-test 우선)은 비용이 낮아 그대로 따랐고, 출시 전 플러그인 모드 라이브 검증으로 Claude Code 2.1.x + macOS에서 exit 2 차단이 정상 동작함을 확인한 뒤 출시했다. 미검증 주장도 대응 비용이 낮으면 따르되, 출시 전 반드시 실측한다는 원칙의 사례.

5. 로드맵에 없던 것들이 운영에서 태어났다

  • freshness 규칙 — "읽었다"는 과거형이다. 읽은 뒤 디스크에서 바뀐 파일은 근거가 아니다 (v0.5).
  • 규칙별 토글 — .grounded/config.json + GROUNDED_DISABLE env. 오탐을 겪은 사용자가 전체 삭제 대신 규칙 하나만 끌 수 있게 (v0.5).
  • 부정 캐시 TTL 10분 — 5분 전에 publish된 패키지가 세션 내내 차단되던 문제 (v0.6).
  • 네트워크 총량 예산 5초 — 조회가 겹치면 최악 ~20초 지연이 가능했다. 예산 초과 시 미캐시 조회는 스킵(fail-open) (v0.5).
§04

시행착오 — 부딪힌 순서대로

① 병렬 hook이 서로의 적립을 지운다 라이브 관측

증상
병렬 Read 4개 중 3개의 근거 적립이 유실되는 것을 라이브로 직접 관측. 드문 경합이 아니라 병렬 도구 호출에서 일상적으로 발생.
원인
hook은 독립 프로세스. load → modify → save가 잠금 없이 겹치면 마지막 저장만 살아남는다.
해결
update_ledger RMW를 fcntl 잠금으로 감쌈. Windows는 msvcrt 영역 잠금, 둘 다 없으면 무잠금 강등. pre_gate의 캐시 저장은 전체 스냅샷 대신 verified_urls/known_pkgs/warned만 병합하는 _save_caches로 — 동시 적립을 덮어쓰지 않게.

② 내장 가드 중복 발견 E2E 검증

증상
미독 파일 Edit 시 pre_gate가 발화하지 않음(마커 부재). Claude Code 내장 read-before-edit가 먼저 거부하고 있었다.
교훈
출시 직전에야 주력 기능이 중복임을 알았다면 포지셔닝이 무너졌을 것. 차별 가치를 실측으로 확인하고 정직하게 재배치(§03-1). 데모와 카피도 G-2 · sed -i 우회 중심으로 다시 짰다.

③ truncate로 만든 파일을 차단하는 오탐 오탐 수정

증상
echo "..." > new.txt로 모델이 직접 만든 파일을 이어서 수정하려 하면 "안 읽었다"고 차단.
원인
셸 truncate 쓰기는 Edit/Write 성공 적립 경로를 타지 않았다.
해결
truncate(>)로 만든 파일은 모델이 전체 내용의 author이므로 근거로 적립. 단 append(>>)와 sed -i는 부분 수정이라 적립하지 않음 — 전체 내용을 안다고 볼 수 없다.

④ heredoc 본문이 파서를 속인다 파싱

증상
cat <<EOF > f의 heredoc 본문 안 텍스트(예: 문서 속 sed -i 예시)가 쓰기 대상/설치 스펙으로 오인.
해결
_mask_heredocs로 본문을 마스킹하고 개행을 세그먼트 분리자에 추가. 정적으로 못 푸는 대상(변수·치환·인용)은 원래부터 통과 — 파서의 한계는 fail-open으로 흡수.

⑤ ledger가 cwd를 따라 표류 v0.6.1 픽스

증상
셸 cd 후 hook payload의 cwd가 바뀌면서 .grounded/ledger.json이 하위 디렉터리들에 흩어짐 — 한쪽에 적립된 근거가 다른 쪽 검문에 안 보임.
해결
ledger 경로를 payload cwd가 아닌 프로젝트 루트에 고정. 출시 태그(v0.6.0) 이후 발견되어 v0.6.1로 즉시 릴리스.

⑥ 방금 publish한 패키지가 계속 차단됨 캐시 정책

증상
G-2 부정 결과(404)를 세션 내내 캐싱 → 사용자가 5분 전 올린 새 패키지가 끝까지 "미존재" 판정.
해결
부정 캐시에만 10분 TTL. 긍정 캐시는 유지(존재하던 패키지가 사라지는 일은 드물다). 네트워크 불명(None)은 애초에 캐싱하지 않아 일시 장애가 자가 치유.

⑦ 봇 차단을 죽은 링크로 오판 설계 단계에서 예방

증상
Cloudflare가 curl을 403으로 막으면 멀쩡한 링크가 "죽은 링크"가 된다.
해결
G-3는 명백한 죽음(404/410/DNS 실패)만 차단, 403/5xx/타임아웃은 경고. HEAD 거부(405/501)는 GET 재시도, curl -X POST/--data는 검사 자체를 스킵(API 엔드포인트는 HEAD를 정당하게 거부), 사설/로컬 호스트 제외(아직 안 띄운 dev 서버는 정상).

⑧ 자기 자신에게 차단당하다 도그푸딩

증상
개발 세션에서 버전 범프를 위해 sed -i를 안 읽은 manifest에 실행 → grounded G-1이 개발자(AI)를 차단.
교훈
차단 메시지대로 파일을 읽고 재시도하니 통과. 강제 루프(차단 → 이유 피드백 → 근거 확보 → 통과)가 실전에서 의도대로 동작한다는 가장 값싼 증거였다.
§05

최종 알고리즘

아키텍처 — 적립과 검문의 분리

hook은 호출마다 독립 프로세스(무상태)이므로 상태는 전부 외부 ledger에 있다. PostToolUse가 적립하고, PreToolUse가 검문한다.

도구 실행됨 (Read / Grep / cat / WebFetch) ──▶ post_record ── 적립 ──▶ .grounded/ledger.json
                                                                  { read_files:    {path: ts},
                                                                    verified_urls: {url: status},
                                                                    known_pkgs:    {eco:name: bool|[bool,ts]},
                                                                    warned:        {rule:target: ts} }
행동 시도 (Edit / Write / Bash / WebFetch) ──▶ pre_gate ── 조회 ──▶ 판정
    PASS → exit 0 조용히 통과
    WARN → exit 0 + additionalContext로 경고 1회 주입
    STOP → exit 2 + stderr 이유가 모델에 피드백 → 모델이 근거 확보 후 재시도

판정 모델 — 오탐은 누락보다 나쁘다

모든 규칙은 세 판정으로 수렴한다. 확신이 없으면 차단이 아니라 경고. 내부 오류는 전부 fail-open: ledger 손상 → 통과, 네트워크 불명 → 통과, 파싱 불가 → 통과. 차단은 근거의 부재가 명백할 때만.

판정조건동작
PASS근거가 ledger에 있음exit 0, 무음
WARN근거 불확실 (403 · 타임아웃 · append · 런타임 결정 대상)exit 0 + 경고 주입 (세션당 규칙:대상별 1회)
STOP근거가 명백히 없음 (미독 · 404 · 레지스트리 미존재)exit 2 + 행동 가능한 이유 피드백

G-1 / G-1s — 파일 쓰기 검문 (의사코드)

gate_file_action(path, ledger, mtime):
    if not exists(path):              return PASS      # 새 파일 생성은 읽을 수 없다
    if path not in ledger.read_files: return STOP      # 근거 없음
    if mtime > read_ts + 1s:          return WARN      # freshness: 읽은 뒤 바뀜 → 다시 읽어라
    return PASS

gate_shell_write(path, mode, ...):                     # Bash 명령에서 shell_scan으로 추출
    mode INPLACE(sed -i, perl -i, tee) | TRUNCATE(>) on unread existing file → STOP
    mode APPEND(>>, tee -a) on unread file             → WARN
    cp/mv onto unread existing file                    → STOP
    find -exec / xargs (대상이 런타임 결정)             → WARN (세션당 1회)
    변수·치환·인용으로 정적 해석 불가                    → PASS (fail-open)

G-2 — 패키지 존재 확인

for (eco, name) in package_specs(command)[:5]:         # pip/uv/npm/pnpm/yarn/cargo 파싱, 명령당 5개
    cached = ledger.known_pkgs[eco:name]               # 부정 캐시는 10분 TTL
    exists = cached if fresh else registry.check(eco, name, timeout=2.5s, budget=5s/command)
    True  → PASS (캐시)   False → STOP "registry에 없음 — 이름을 검색해 확인하라" (캐시)
    None  → PASS 무음 (네트워크 불명은 환각의 증거가 아니다 · 캐싱 안 함)

G-3 — URL 생존 확인

for url in fetch_urls(command)[:3] | WebFetch.url:     # POST/--data는 스킵, 사설호스트 스킵
    status = HEAD(url) retry GET on 405/501            # fragment 정규화
    2xx/3xx → PASS   404/410/DNS(0) → STOP   403/5xx/timeout → WARN (봇차단일 수 있다)
    확정 결과만 캐싱 — 모호한 결과는 자가 치유를 위해 비캐싱

경고 멱등화

_claim_warns(ledger, warns):
    key = rule:target                                  # freshness는 rule:path:int(mtime) — 새 변경엔 재경고
    이미 warned에 있으면 드롭, 없으면 기록 후 주입
    주입 시 말미에 1줄: "advisory · 이 세션에서 반복되지 않음 · 지우려고 재시도하지 마라"
    STOP과 동시 발생 시 경고는 claim하지 않음            # 차단 해소 후 전달 기회 보존

참고 문헌 · 기술 근거

연구 문헌

  • Spracklen, J., Wijewickrama, R., Sakib, A.H.M.N., Maiti, A., Viswanath, B., Jadliwala, M. — "We Have a Package for You! A Comprehensive Analysis of Package Hallucinations by Code Generating LLMs", USENIX Security 2025. G-2의 동기: 576,000개 생성 샘플에서 LLM 추천 패키지의 5.2–21.7%(고유 이름 205,474개)가 실존하지 않음을 측정 — "slopsquatting" 공격면.
  • "AgentSpec: Customizable Runtime Enforcement for Safe and Reliable LLM Agents", ICSE 2026. 에이전트 안전을 모델 밖의 런타임 강제(trigger → predicate → enforcement)로 정식화. grounded는 이 모델을 Claude Code hook 경계에 특화한 형태(matcher → ledger 판정 → exit code).
  • "Defeating Prompt Injections by Design (CaMeL)", Google DeepMind. LLM을 신뢰 불가 구성요소로 두고 결정론적 계층이 통제권을 갖는 설계 스탠스. CaMeL은 프롬프트 주입에, grounded는 근거 없는 행동에 같은 스탠스를 적용.

기술 문서 (구현 전 전수 검증)

  • Claude Code Hooks 공식 문서 — 핵심 주장 검증: PreToolUse exit 2 = 도구 차단 + stderr 모델 피드백 ✓ · PostToolUse exit 2는 차단 불가(이미 실행됨) ✓ · hook은 호출마다 독립 프로세스 ✓ · tool_input은 stdin JSON ✓. 이 검증이 외부 ledger 설계의 정당성 근거.
  • 레지스트리 API — npm registry.npmjs.org/{name} · PyPI pypi.org/simple/{name}/ · crates.io crates.io/api/v1/crates/{name} (UA 헤더 필수). 전부 실조회 라이브 검증.
  • Slopsquatting — 환각 패키지명을 선점 등록하는 공급망 공격의 명명.

수요의 실증 (독립적으로 같은 문제를 제기한 열린 이슈)

  • opencode #27901 — edit 도구의 read-before-edit가 설명문에만 있고 런타임 강제가 없음을 발견, 강제 옵션 요청. "설명문은 제안, 강제는 코드"라는 본 프로젝트의 핵심 명제와 동일.
  • claude-code #65982 — "검증 행동이 선행됐는지 확인하는 게이트" 기능 요청. grounded가 도구 경계에서 하는 것의 텍스트 단언 버전.
  • qwen-code #4939 — grep을 read-before-edit 근거로 인정해달라는 요청. grounded의 근거원 확대 결정과 독립 수렴.

영구 제외 (만들지 않기로 한 것)

  • LLM 기반 문장 단위 사실 검증 — 판정에 LLM을 넣는 순간 결정론이 깨진다. 이 프로젝트의 정체성과 모순.
  • 대시보드 · 통계 · 텔레메트리 — 가드레일은 조용해야 한다.

정직한 한계 (약속하지 않는 것)

도구 경계의 행동에만 근거를 강제한다. 모델이 도구 없이 채팅 텍스트로 주장을 뱉으면 어떤 hook도 보지 못한다(SessionStart 프롬프트 룰 주입으로 부분 완화만). 흔한 쓰기 관용구를 탐지하되 적대적 우회는 탐지 경계 밖이다. 모든 환각의 차단을 약속하지 않는다 — 근거 없는 행동의 차단을 약속한다.
