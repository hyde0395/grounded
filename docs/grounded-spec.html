grounded.

"확인했다"고 말하기 전에, 정말 확인하게 만든다. AI 코딩 에이전트가 근거 없이 행동하려 할 때 결정론적으로 멈추는 가드레일.

파일을 읽지 않고 고치려 하거나, 존재를 확인하지 않은 패키지를 설치하려 하거나, 살아있는지 모르는 URL을 들이밀 때 — 그 행동이 실행되기 전에 검문한다. LLM 없이, 셸과 로컬 상태만으로.

01

왜 만드는가

AI 코딩 에이전트의 가장 비싼 실패는 "위험한 명령"이 아니라 "확인하지 않은 것을 확인한 척하는 것"이다.

에이전트는 종종 파일을 읽지 않고 내용을 추측해 수정하고, 존재하지 않는 패키지를 설치하려 하고, 죽은 링크를 "검색해보니 있다"며 제시한다. 각각은 작아 보이지만, 잘못된 전제 위에 다음 행동이 쌓이면서 사람이 그것을 일일이 되짚어 검증해야 하는 피로가 쌓인다. 그 피로가 진짜 비용이다.

핵심 통찰

CLAUDE.md에 "추측하지 마라"라고 적는 것은 LLM이 런타임에 해석하는 제안이다. 설득당할 수 있다. 반면 hook은 모델이 무슨 생각을 하든 무조건 실행되는 강제다. 근거 없는 행동을 막으려면 제안이 아니라 강제가 필요하다.

grounded는 "더 똑똑한 모델"로 이 문제를 풀지 않는다. 행동 직전에 결정론적으로 근거를 요구한다. 파일을 고치려면 먼저 읽었어야 하고, 패키지를 깔려면 먼저 존재를 확인했어야 한다. 근거가 세션 기록에 없으면, 그 행동은 통과하지 못한다.

02

이미 있는 것과 무엇이 다른가

PreToolUse 가드레일 자체는 흔하다. 그러나 전부 "보안"이다. grounded는 다른 축에 선다.

축기존 가드레일 (다수)grounded
막는 대상위험한 행동 (rm -rf, force push, .env 유출, 시크릿)근거 없는 행동 (안 읽고 수정, 미확인 설치, 죽은 링크)
질문"이 명령이 위험한가?""이 행동에 근거가 있는가?"
판단 기준명령어 패턴 (정적 정규식)세션 행동 이력 (동적 상태)
상태대체로 무상태 (패턴만 보면 됨)상태 필수 (무엇을 읽었나 기억해야 함)
시장레드오션 (다수 경쟁)빈 칸
포지셔닝 규칙

"종합 보안 가드레일"로 포지셔닝하면 기존 강자들과 정면 충돌한다. grounded는 반드시 "AI가 추측을 사실처럼 말하는 것을 막는다"는 별도 카테고리로 선을 긋는다. 보안 도구와 함께 쓰는 보완재이지, 대체재가 아니다.

03

아키텍처

hook은 호출마다 독립 프로세스로 떠서 상태가 없다. 그래서 외부 원장(ledger)이 코어가 된다.

두 개의 hook이 하나의 로컬 상태 파일을 공유한다. PostToolUse가 적립하고, PreToolUse가 검문한다.

  ┌─────────────────────── 세션 동안 누적 ───────────────────────┐

  도구 실행됨 (Read/Grep/web_fetch/검증성공)
        │
        ▼
  PostToolUse hook ──적립──▶  .grounded/ledger.json
        │                          { read_files: [...],
   (관찰·기록 전용)                  verified_urls: [...],
                                    known_pkgs: [...] }
                                          │
  ─────────────────────────────────────── │ ───────────────
                                          │ 조회
  AI가 행동 시도 (Edit/Write/install/fetch)│
        │                                 ▼
        ▼                          PreToolUse hook
  tool_input 가로챔 ───────────▶  원장에 근거 있나?
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
              근거 있음        불확실          근거 없음
              exit 0           exit 0 +        exit 2
              (통과)           경고 주입        (차단·피드백)

#구성 요소

요소역할비고
ledger.json이번 세션에서 확인된 사실 누적 (읽은 파일, 검증된 URL, 확인된 패키지)SessionStart에서 초기화
post-recordPostToolUse — 도구 결과에서 "무엇이 확인됐나" 추출해 원장에 적립관찰 전용, 차단 안 함
pre-gatePreToolUse — 행동의 대상이 원장에 근거가 있는지 대조 후 판정exit 2로 차단 가능
verdictpass / warn / stop 결정 로직 (순수 함수, LLM 없음)규칙별로 다름
검증된 기술 제약

① hook은 매번 독립 프로세스 → 상태는 반드시 파일/데몬으로. ② PreToolUse는 exit 2로 차단 + stderr가 모델에 피드백된다. ③ 단, 버전·배포 방식에 따라 차단이 무시된 버그 이력이 있다(플러그인 command hook, 일부 macOS 버전). 그래서 플러그인 패키징보다 settings.json 직접 등록으로 시작하고, 첫 줄 테스트로 exit 2가 실제로 막는지부터 확인한다.

04

규칙 셋

세 가지 근거 규칙. 결정론·오탐·사각지대 기준으로 우선순위가 다르다. 단단한 것부터 출시한다.

G-1

읽기 선행 — 안 읽은 파일은 수정 못 한다

MVP
대상: Edit / Write · 근거원: 이번 세션의 Read / Grep / cat · 오탐: 매우 낮음
Edit(foo.py) 시도→ 원장에 foo.py 읽은 기록?→ 없음 → 차단

가장 단단한 규칙. 대상이 로컬 파일 경로라 네트워크·오탐이 없고, Edit/Write는 무조건 도구를 거치므로 사각지대도 없다. "내용을 추측해서 고치는" 가장 흔한 실패를 정확히 막는다.

차단 메시지 예

"foo.py를 이번 세션에 읽은 기록이 없습니다. 추측으로 수정하지 말고 먼저 파일을 읽으세요."

G-2

존재 확인 — 미확인 패키지는 설치 못 한다

NEXT
대상: Bash (npm/pip/cargo install) · 근거원: 레지스트리 존재 확인 · 오탐: 낮음
pip install foopkg→ PyPI에 foopkg 존재?→ 없음 → 차단

"환각 패키지 설치"를 막는다 — 존재하지 않는 이름을 지어내 설치하려는 흔한 실패. 레지스트리 메타데이터 조회(HEAD/JSON)는 일반 웹페이지보다 안정적이라 봇 차단 오탐이 적다. 결과는 원장에 캐싱한다.

G-3

링크 실존 — 핑 안 때린 URL은 경고한다

LATER
대상: web_fetch / Bash(curl) · 근거원: HTTP 상태 · 오탐: 높음 → soft block
fetch(url)→ 404/DNS실패→ 차단 403/타임아웃→ 경고만

가장 까다로운 규칙이라 마지막. 죽은 링크(404·DNS 실패)는 명백한 환각이니 차단하되, 403·5xx·타임아웃은 봇 차단일 수 있어 차단하지 않고 경고만 던진다. 오탐으로 작업을 끊는 것보다 1%의 통과가 낫다.

사각지대 주의

AI가 도구 없이 일반 텍스트로 링크를 뱉으면 hook을 타지 않는다. 이건 hook만으로 못 막는다 → 09 로드맵의 프롬프트 룰로 보완.

05

판정 모델 — pass / warn / stop

모든 규칙은 세 결과 중 하나로 수렴한다. retention을 지키는 핵심은 "언제 stop이 아니라 warn인가"이다.

판정언제동작사용자 체감
PASS근거가 원장에 있음exit 0 조용히 통과아무 방해 없음
WARN근거 불확실 (403, 타임아웃, 모호)exit 0 + 모델에 경고 주입흐름 안 끊김, 모델이 스스로 재확인
STOP근거 명백히 없음 (안 읽음, 404, 미존재)exit 2 차단 + 이유 피드백막힘 — 단, 이유와 해법 제시
설계 철학 — 오탐 < 누락

개발자 도구는 "내 시간을 아껴주는가"로 살고 죽는다. 멀쩡한 작업을 잘못 막으면(false positive) 사용자는 10분 만에 끈다. 그래서 확신이 없으면 STOP이 아니라 WARN이다. 1%의 환각을 통과시키더라도, 오탐으로 흐름을 끊지 않는 쪽을 택한다. 차단은 "근거가 명백히 없을 때"로만 좁힌다.

06

세션 원장

hook이 무상태이므로, "이번 세션에서 무엇이 확인됐나"를 담는 단일 진실 공급원.

// .grounded/ledger.json — SessionStart에서 {} 로 초기화
{
  "read_files": {
    "src/auth.py": 1718000000,   // 읽은 시각 (이후 외부 수정 감지용)
    "README.md":   1718000042
  },
  "verified_urls": {
    "https://docs.x.com/api": 200,   // 캐싱: 재검증 안 함
    "https://dead.example/x":  404
  },
  "known_pkgs": {
    "requests": true,
    "reqests":  false     // 오타/환각 패키지
  }
}

#적립 규칙 (PostToolUse)

  • Read / Grep / cat 성공 → 해당 파일 경로를 read_files에 시각과 함께 적립
  • web_fetch / curl 성공 → 그 URL과 상태코드를 verified_urls에 적립
  • 패키지 조회 성공 → 패키지명을 known_pkgs에 적립
신선도(freshness) 처리

읽은 시각을 기록하는 이유: 파일을 읽은 뒤 다른 도구가 그 파일을 수정하면, 원장의 "읽음"은 낡은 것이 된다. G-1은 "수정 시점 기준으로 최신 읽기가 있는가"까지 볼 수 있다(2차 정밀화). MVP에서는 단순히 "읽은 적 있는가"로 시작한다.

07

사각지대 — 정직하게

hook으로 못 잡는 것을 README에 숨기지 않고 먼저 밝힌다. 이게 신뢰를 만든다.

사각지대 1 — 텍스트 응답

AI가 도구를 쓰지 않고 채팅 텍스트로 "이 링크 참고하세요: https://..." 라고 뱉으면 어떤 hook도 타지 않는다. hook은 도구 호출만 가로챈다. 이건 hook의 구조적 한계이며, 프롬프트 룰(09)로만 부분 완화된다.

사각지대 2 — 봇 차단 오탐

Cloudflare 등이 curl을 403으로 막으면 멀쩡한 링크를 죽은 것으로 오판할 수 있다. 그래서 G-3은 403을 차단하지 않고 경고로만 처리한다(05 판정 모델). 완벽한 URL 검증은 불가능하다는 것을 인정하고 설계에 반영한다.

그래서 — 무엇을 약속하고 무엇을 약속하지 않는가

약속한다: 도구를 거치는 행동(수정·설치·fetch)에 대해 근거를 강제한다. 약속하지 않는다: 모든 환각을 잡는 것. grounded는 "AI의 똥을 자동으로 다 치운다"가 아니라 "가장 흔하고 비싼 근거 없는 행동을 도구 경계에서 막는다"이다.

08

MVP 범위

전부 만들지 않는다. 가장 단단한 규칙 하나로 시장 반응을 싸게 확인한다.

#v0.1 — G-1 단일 규칙

  1. exit 2 작동 확인 — 1줄 hook으로 "내 환경에서 PreToolUse가 진짜 도구를 막는가" 먼저 검증. 안 되면 HTTP hook 방식으로 전환.
  2. ledger 초기화 — SessionStart hook이 .grounded/ledger.json을 {}로 생성
  3. 적립 — PostToolUse가 Read/Grep 성공 시 파일 경로 기록
  4. 검문 — PreToolUse가 Edit/Write 대상이 원장에 있나 대조, 없으면 exit 2 + 메시지
  5. settings.json 등록 — 플러그인 패키징 없이 직접 등록 (command hook 버그 회피)
왜 G-1부터인가

G-1은 오탐 0에 가깝고(로컬 파일), 사각지대 0(Edit는 무조건 도구), 네트워크 의존 0(지연·봇차단 없음). 즉 세 규칙 중 유일하게 "조용히 정확하게" 작동한다. 첫인상이 오탐으로 망가지지 않는다.

#의도적으로 뺀 것

  • G-2/G-3 — v0.1 반응 본 뒤 추가
  • LLM 기반 문장 단위 사실 검증 — 지연·비용·오탐으로 "아름다운 쓰레기"가 됨. 영구 제외.
  • 대시보드·통계 — 기존 보안 가드레일의 영역. 근거 강제에 집중.
